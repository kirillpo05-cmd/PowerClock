# SPEC.md — PowerClock, техническая спецификация

> **Слой 2** по методологии Spec-First. Сгенерирован из `PROJECT_IDEA_PowerClock.md`.
> Модули в порядке сборки (из «Плана запуска»): `pipeline` → `clock` → `insights` →
> `weekring` → `spiral`. Витрина (shell/дизайн) — сквозной раздел в конце.

## Глобальные решения и допущения

- **API отдаёт 15-минутные интервалы** (Nord Pool MTU-15: 96 точек/сутки, поле
  `timestamp` в Unix-секундах UTC, `price` в EUR/MWh). Пайплайн усредняет 4 слота
  в час: `price(hour) = mean(слоты, попавшие в этот местный час)`. *(допущение —
  проверено живым запросом к API 2026-07-05)*
- **Таймзона.** Всё, что называется «час» и «дата» в данных и UI — **Europe/Riga**
  (EET/EEST). Конверсия из UTC — только через `Intl.DateTimeFormat` с
  `timeZone: 'Europe/Riga'` (Node ≥ 18 и все браузеры), никаких ручных `+2/+3`.
- **Схема daily расширена на 3 зоны** (`LV`, `EE`, `LT`) — v1.0 требует переключатель
  зон, данные приходят одним ответом API. *(допущение — идея описывала только LV)*
- **Бэкофилл 400 дней** вместо 60: иначе Year Spiral пуста на 9 месяцев из 12.
  Окно для weekday×hour-агрегатов остаётся **60 дней**, `by_month` считается по всей
  доступной истории. *(допущение)*
- **`window_avg`** (средняя цена за окно 60 дней) добавлено в aggregates — нужно
  инсайту «сегодня на N% дороже среднего». *(допущение)*
- Канонические данные лежат в `data/` в корне репо (как в идее). Vite-плагин
  раздаёт их в dev-сервере и копирует в `dist/data` при сборке.
- Деплой — GitHub Pages, `base: '/PowerClock/'` в Vite; все fetch-пути строятся от
  `import.meta.env.BASE_URL`.
- Цены хранятся в **EUR/MWh**, показываются в **ц/кВт·ч** (`÷10`), 1 знак после
  запятой. Единственная точка конверсии — `toCtKwh()` в `src/lib/scales.ts`.
- Отрицательные цены возможны (бывают на NPS) — шкала и бары обязаны их переживать
  (см. крайние случаи `clock`).

## Модуль 1: `pipeline`

Скрипт `scripts/fetch_prices.mjs` (Node, без зависимостей) + workflows
`.github/workflows/data.yml`, `.github/workflows/deploy.yml`.

### 1.1 User Stories

- Как **владелец сайта**, я хочу, чтобы данные обновлялись сами раз в день, чтобы
  сайт жил без моего участия.
- Как **посетитель**, я хочу видеть завтрашние цены вечером того же дня, чтобы
  спланировать стирку/зарядку.
- Как **разработчик**, я хочу одним прогоном скрипта загрузить историю за N дней,
  чтобы наполнить агрегаты перед первым деплоем.
- Как **разработчик**, я хочу, чтобы фронт зависел только от моей JSON-схемы,
  чтобы смена формата Elering чинилась в одном скрипте.
- Крайний случай: как **владелец**, я хочу, чтобы при недоступности API workflow
  падал с внятной ошибкой (а не коммитил пустые файлы), чтобы старые данные
  оставались целыми.

### 1.2 Модель данных (JSON-схемы)

```jsonc
// data/daily/YYYY-MM-DD.json  — один местный (Europe/Riga) день
{
  "date": "2026-07-05",          // string, YYYY-MM-DD, местная дата
  "unit": "EUR/MWh",             // string, константа
  "zones": {                      // объект, ключи ровно "LV" | "EE" | "LT"
    "LV": [ { "hour": 0, "price": 21.4 }, /* …обычно 24 */ ],
    "EE": [ /* … */ ],
    "LT": [ /* … */ ]
  }
}
// hour: integer 0–23, местный час; price: number, EUR/MWh, округление до 0.01.
// В день перехода на летнее время записей 23 (нет часа 03); в день перехода на
// зимнее — 24, «двойной» час усреднён по 8 слотам.
```

```jsonc
// data/aggregates.json
{
  "updated_at": "2026-07-05T13:04:00Z",  // ISO-8601 UTC, время генерации
  "window_days": 60,                      // integer, окно weekday×hour и window_avg
  "zones": {
    "LV": {
      "by_weekday_hour": [ [ /* 24 числа или null */ ], /* …7 строк, Mon=0 */ ],
      "by_month": [ { "month": 1, "avg": 96.2 }, /* …12; avg: number|null */ ],
      "window_avg": 87.3                  // number|null, средняя за окно
    },
    "EE": { /* так же */ },
    "LT": { /* так же */ }
  }
}
```

### 1.3 API и серверная логика

Внешний API (только в пайплайне, из браузера НЕ вызывается):

- `GET https://dashboard.elering.ee/api/nps/price?start={ISO-UTC}&end={ISO-UTC}`
  → `200 {"success": true, "data": {"ee": [{"timestamp": 1783123200, "price": 18.83}, …], "lv": […], "lt": […], "fi": […]}}`
  Ошибки: не-200, `success !== true`, пустой `data.lv` → скрипт завершает процесс
  с exit code 1 и сообщением в stderr; ничего не пишет на диск.

CLI скрипта:

```
node scripts/fetch_prices.mjs                  # диапазон по умолчанию: вчера…завтра (Riga)
node scripts/fetch_prices.mjs --from 2025-06-01 --to 2026-07-05   # бэкофилл
```

Алгоритм: местный диапазон дат → UTC-интервал с запасом ±1 день → 1 запрос к API
(при бэкофилле — чанками по 90 дней, чтобы не упереться в лимит ответа) →
группировка слотов по (зона, местная дата, местный час) → усреднение → запись
`data/daily/*.json` **только для полных дней** (у LV ≥ 23 часов; неполный «завтра»
не пишется — допишется завтрашним прогоном) → пересчёт `data/aggregates.json`
по всем daily-файлам.

Workflows:

- `data.yml`: `schedule: cron '0 13 * * *'` (13:00 UTC = 15:00 EET / 16:00 EEST —
  после публикации day-ahead ≈13:45 EET) + `workflow_dispatch` с необязательными
  inputs `from`, `to`. Шаги: checkout → setup-node 22 → `node scripts/fetch_prices.mjs`
  → `git add data && git commit || echo "no changes"` → push. Права: `contents: write`.
  После успешного коммита вызывает deploy через `workflow_call`.
- `deploy.yml`: `on: push (main) + workflow_dispatch + workflow_call`. Шаги:
  checkout (`ref: main`) → setup-node → `npm ci` → `npm run build` (включает копию
  `data/` в `dist/data`) → `actions/upload-pages-artifact` → `actions/deploy-pages`.
  Права: `pages: write`, `id-token: write`.

### 1.4 Экраны и компоненты

Нет UI. Наблюдаемость: логи workflow (`fetched 3 days, wrote 2 files, aggregates
updated`), бейдж «updated N h ago» на фронте (из `updated_at`).

### 1.5 Бизнес-логика

- Средняя часа = среднее арифметическое всех слотов, чей `timestamp` попадает в
  этот местный час (обычно 4; на DST-стыке 0 или 8).
- Округление цен в файлах: 2 знака (EUR/MWh).
- `by_weekday_hour[w][h]` = среднее `price` по всем daily-файлам за последние
  `window_days` **завершённых** дней (сегодня не включается — день неполный по
  смыслу потребления; завтра тем более), где weekday(date, Riga) == w (Mon=0).
  Если наблюдений нет — `null`.
- `by_month[m].avg` = среднее по всем daily всех лет с месяцем m+1; `null` если пусто.
- `window_avg` = среднее по всем часам окна.
- Идемпотентность: повторный прогон на тех же данных даёт байт-в-байт те же файлы
  (сортировка ключей и дат детерминированная) → пустой diff → нет лишних коммитов.

### 1.6 Крайние случаи

| Сценарий | Ожидаемое поведение |
|---|---|
| API недоступен / 5xx / таймаут 30 с | exit 1, файлы не тронуты, workflow красный |
| `success: false` или нет `data.lv` | exit 1 с телом ответа в stderr |
| Завтрашние цены ещё не опубликованы | файл завтра не создаётся; фронт показывает «tomorrow — not published yet» |
| DST весна (23 часа) | файл с 23 записями, часа 03 нет |
| DST осень (25 часов слотов) | 24 записи, двойной час усреднён |
| Частично опубликованный день (< 23 ч у LV) | день пропускается с warning в лог |
| Бэкофилл поверх существующих файлов | файлы перезаписываются идемпотентно |
| В ответе есть лишние зоны (fi) | игнорируются |

## Модуль 2: `clock` (Price Clock — главный экран)

`src/charts/ClockChart.tsx` + `src/lib/loader.ts`, `src/lib/scales.ts`.

### 2.1 User Stories

- Как **посетитель**, я хочу с одного взгляда видеть дешёвые и дорогие часы
  сегодняшних суток, чтобы решить, когда включать технику.
- Как **посетитель**, я хочу видеть, какой час идёт сейчас и сколько он стоит,
  чтобы понять «дорого ли прямо сейчас».
- Как **посетитель**, я хочу переключиться на завтра (когда цены опубликованы),
  чтобы спланировать вечер/утро.
- Как **посетитель**, я хочу навести курсор на любой час и увидеть точную цену,
  чтобы не гадать по цвету.
- Крайний случай: как **посетитель ночью до публикации**, я хочу видеть явную
  надпись, что завтрашних цен ещё нет, а не пустой круг.

### 2.2 Модель данных

Вход — `DayPrices` из loader:

```ts
type HourPrice = { hour: number; price: number };        // EUR/MWh
type DayPrices = { date: string; zones: Record<'LV'|'EE'|'LT', HourPrice[]> };
type ClockProps = {
  day: DayPrices; zone: Zone;            // Zone = 'LV'|'EE'|'LT'
  currentHour: number | null;            // null, если показываем не сегодня
  colorScale: (eurMwh: number) => string;
  maxPrice: number;                      // общий максимум сегодня+завтра — единый масштаб
};
```

### 2.3 API

Нет сетевых вызовов из компонента. `loader.ts`:

- `loadDaily(date: string): Promise<DayPrices | null>` —
  `fetch(BASE_URL + 'data/daily/' + date + '.json')`; 404 → `null`; сетевые ошибки →
  `null` + `console.warn`.
- `loadAggregates(): Promise<Aggregates | null>` — аналогично.
- `todayRiga(): string`, `tomorrowRiga(): string`, `currentHourRiga(): number` —
  через `Intl.DateTimeFormat('en-CA'/'…', { timeZone: 'Europe/Riga' })`.

### 2.4 Экраны и компоненты

- SVG-«циферблат»: 24 радиальных бара, **00:00 сверху, по часовой**; длина бара
  и цвет кодируют цену (правило идеи: длина + цвет, не только цвет). Внутренний
  радиус ~38% внешнего.
- Метки часов 00/03/06/09/12/15/18/21 снаружи; в центре — крупно текущая цена
  «10.9 ct/kWh» + подпись «now» (только на вкладке Today).
- Текущий час: бар обведён акцентом `#60A5FA` + тонкая «стрелка» от центра.
- Переключатель **Today / Tomorrow** (сегменты-кнопки). Tomorrow disabled +
  подпись «published ~15:30», пока файла нет.
- Состояния: **loading** — скелетон-круг с пульсацией; **error/empty** (нет и
  сегодняшнего файла) — карточка «Data unavailable, try later» + причина;
  **success** — график с анимацией входа: бары растут от 0 до длины 500 мс ease-out,
  каскад 15 мс/бар. Tooltip: «07:00–08:00 · 12.4 ct/kWh» — следует за курсором,
  показывается мгновенно.

### 2.5 Бизнес-логика

- Угол: `angle(hour) = hour/24 * 2π - π/2` (начало сектора), ширина сектора
  `2π/24` минус зазор 0.012 рад.
- Длина: линейная шкала `[min(0, minPrice) … maxPrice] → [0 … R-r0]`. Единый
  `maxPrice` для today/tomorrow, чтобы переключение честно сравнивало дни.
- Цвет: `d3.scaleSequential` с интерполятором по `#22C55E → #EAB308 → #EF4444`,
  домен — [p10, p90] окна 60 дней из aggregates (робастность к выбросам); клэмп.
  Если aggregates нет — домен по текущему дню.
- Текущий час подсвечивается только если выбранная вкладка = сегодня и зона любая.
- Переключение зоны (из shell) меняет данные без перезагрузки; анимация повторно
  не проигрывается (только при первом маунте).

### 2.6 Крайние случаи

| Сценарий | Поведение |
|---|---|
| Нет файла за сегодня (cron упал > суток назад) | показать последний доступный день из aggregates нельзя — показываем error-состояние + бейдж «updated N h ago» краснеет |
| Нет завтрашнего файла | вкладка Tomorrow disabled с подписью |
| 23 записи (DST) | отсутствующий час рендерится «дыркой», tooltip его пропускает |
| Отрицательная цена | длина бара по шкале от min(0, minPrice); цвет — минимум шкалы; в tooltip знак минус |
| Все цены одинаковые | шкала длины вырождается — задать минимальную длину бара 4 px |
| Узкий экран (< 420 px) | SVG скейлится viewBox'ом, центр-цифра ≥ 20 px |
| Полночь во время просмотра | `currentHourRiga()` пересчитывается таймером раз в минуту; в 00:00 «сегодня» станет вчерашним файлом → loader перезапрашивает daily |

## Модуль 3: `insights`

`src/lib/insights.ts` (чистые функции) + `src/ui/Card.tsx`.

### 3.1 User Stories

- Как **посетитель**, я хочу готовый вывод «самый дешёвый час сегодня — 03:00
  (2.1 ц)», чтобы не разглядывать график.
- Как **посетитель**, я хочу знать самый дорогой час, чтобы его избежать.
- Как **житель**, я хочу знать самое дешёвое время недели, чтобы планировать
  энергоёмкие дела заранее.
- Как **посетитель**, я хочу видеть, дороже ли сегодня обычного, чтобы иметь контекст.
- Крайний случай: как **посетитель** при неполных данных, я хочу видеть карточки
  только с достоверными выводами (лишние скрыты), а не «NaN%».

### 3.2 Модель данных

```ts
type Insight = {
  id: 'cheapest-today' | 'priciest-today' | 'best-week-slot' | 'today-vs-avg';
  title: string;        // "Cheapest hour today"
  value: string;        // "03:00–04:00"
  detail: string;       // "2.1 ct/kWh"
  tone: 'good' | 'bad' | 'neutral';
};
```

### 3.3 API

Чистые функции, без сети:

```ts
buildInsights(today: HourPrice[] | null, agg: ZoneAggregates | null,
              currentHour: number | null): Insight[]
// внутри: cheapestHour(), priciestHour(), bestWeekSlot(), todayVsWindow()
```

### 3.4 Экраны и компоненты

Ряд из ≤ 4 карточек под часами (grid, на мобиле — колонка). Карточка: заголовок
(caps, 12 px, приглушённый), значение (28 px, tabular-nums), деталь. `tone`
подкрашивает значение (good `#22C55E`, bad `#EF4444`, neutral — текст). Состояния:
loading — скелетоны; данных для карточки нет — карточка не рендерится (ряд сжимается).

### 3.5 Бизнес-логика

- `cheapest/priciest`: min/max по `price`; при равенстве — меньший час. Формат
  «HH:00–HH+1:00», цена в ц/кВт·ч (1 знак).
- `best-week-slot`: argmin по `by_weekday_hour` (игнорируя null); формат
  «Sun 03:00»; требует ≥ 50% заполненных ячеек, иначе карточка скрывается.
- `today-vs-avg`: `(mean(today) - window_avg) / window_avg * 100`, округление до
  целого; |Δ| < 3% → «about average» (tone neutral); дороже → bad «+18% vs 60-day
  avg», дешевле → good.
- Все функции тотальны: на `null`/пустой массив возвращают «нет инсайта», не бросают.

### 3.6 Крайние случаи

| Сценарий | Поведение |
|---|---|
| `today == null` | остаются только карточки из агрегатов (best-week-slot) |
| `window_avg == null` или 0 | карточка today-vs-avg скрыта (деление на 0 исключено) |
| Все цены дня равны | cheapest и priciest показывают одинаковую цену — допустимо |
| Отрицательная средняя окна | процентное сравнение скрыто (бессмысленно), остальное живёт |
| aggregates старше 48 ч | карточки из агрегатов показываются, но shell красит бейдж свежести |

## Модуль 4: `weekring` (Weekly Ring)

`src/charts/WeekRing.tsx`.

### 4.1 User Stories

- Как **посетитель**, я хочу увидеть паттерн недели (дешёвая ночь, дорогой вечер
  будня) одним взглядом, чтобы понять структуру цен.
- Как **житель**, я хочу сравнить выходные с буднями, чтобы переносить дела на
  выходные осознанно.
- Как **посетитель**, я хочу tooltip с точным значением ячейки, чтобы проверить
  впечатление от цвета.
- Крайний случай: как **посетитель** при куцей истории, я хочу видеть, какие
  ячейки не имеют данных (нейтральный цвет), а не враньё.

### 4.2 Модель данных

Вход: `by_weekday_hour: (number|null)[7][24]` выбранной зоны + `window_days`.

### 4.3 API

Нет. Данные приходят props'ами из App (loader уже отработал).

### 4.4 Экраны и компоненты

- SVG: 7 концентрических колец × 24 сектора; **Mon — внутреннее кольцо**, Sun —
  внешнее; 00:00 сверху, по часовой. Зазор между кольцами 2 px, между секторами
  0.008 рад.
- Снаружи метки часов (00/06/12/18), слева от колец — буквы дней (M T W T F S S).
- Легенда-градиент под графиком: min → max ц/кВт·ч (общий компонент `Legend`).
- Tooltip: «Tue 18:00–19:00 · avg 14.2 ct/kWh (60 d)».
- Состояния: loading — скелетон; aggregates нет — блок скрыт целиком (страница
  живёт без него); success — вход: кольца проявляются opacity 0→1 по 60 мс каскадом.

### 4.5 Бизнес-логика

- Цветовая шкала — **та же** sequential-шкала, что у clock (единый язык цвета),
  домен — [p10, p90] значений матрицы.
- `null`-ячейка: заливка `#1B2233` (нейтральная), tooltip «no data».
- Подпись под заголовком: «average price by weekday × hour, last 60 days»
  (число — из `window_days`, не хардкод).

### 4.6 Крайние случаи

| Сценарий | Поведение |
|---|---|
| Вся матрица null | блок скрыт |
| < 7 дней истории (часть ячеек null) | null-ячейки нейтральные, легенда по имеющимся |
| Все значения равны | домен шкалы [v-1, v+1], середина градиента |
| Ресайз окна | viewBox, без перерисовки данных |

## Модуль 5: `spiral` (Year Spiral)

`src/charts/YearSpiral.tsx`.

### 5.1 User Stories

- Как **посетитель**, я хочу видеть сезонность (дорогая зима / дешёвое лето),
  чтобы понять годовой контекст цен.
- Как **рекрутёр**, я хочу увидеть необычную, но читаемую визуализацию, чтобы
  оценить вкус автора.
- Как **посетитель**, я хочу tooltip по месяцу с точным средним, чтобы уточнить цвет.
- Крайний случай: как **посетитель**, я хочу видеть, какие месяцы без данных
  (приглушены), чтобы не делать ложных выводов.

### 5.2 Модель данных

Вход: `by_month: { month: 1–12; avg: number|null }[12]` выбранной зоны.

### 5.3 API

Нет — props из App.

### 5.4 Экраны и компоненты

- SVG-спираль: 12 дуг по архимедовой спирали, январь начинается сверху, ход по
  часовой, радиус растёт от центра наружу (~1.2 витка). Толщина дуги ~18 px,
  цвет — средняя месяца по общей шкале; месяц с `avg: null` — нейтральный
  `#1B2233`, приглушённая метка.
- Метки месяцев (Jan…Dec, 11 px) снаружи каждой дуги.
- Tooltip: «March · avg 11.3 ct/kWh». Легенда общая с weekring.
- Состояния: loading — скелетон; все null — блок скрыт; success — дуги
  «прорисовываются» (stroke-dashoffset) 600 мс ease-out.

### 5.5 Бизнес-логика

- Полярные координаты: `θ(m) = m/12 * 2π - π/2`, `r(m) = r0 + (R - r0) * m/12`,
  дуга от θ(m) до θ(m+1) по спирали (сегменты полилинии, шаг 4°).
- Кодирование значения — **только цвет** (радиус здесь — ось времени, не значение);
  это осознанное исключение из правила «длина+цвет», компенсируется легендой и
  tooltip. *(допущение, следует правилу идеи «сравнение только по углу/цвету, не
  по радиусу»)*
- Домен шкалы — [min, max] ненулевых `avg`.

### 5.6 Крайние случаи

| Сценарий | Поведение |
|---|---|
| Данных < 12 месяцев (первый год жизни) | пустые месяцы нейтральные |
| Все null | блок скрыт |
| Один месяц с данными | домен [v-1, v+1], дуга средним цветом |
| Экстремальный месяц (энергокризис) | клэмп не применяем — спираль показывает реальный диапазон |
