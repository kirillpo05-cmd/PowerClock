# SPEC-FIRST: рабочая инструкция (по методичке, с готовыми промптами)

Практический плейбук: как провести ЛЮБОЙ проект по 5 слоям методологии
**Идея → Спецификация → Конфигурация → Пакет → Сборка** с Claude Code.
Каждый шаг: что делаешь ты → какой промпт копируешь → что проверяешь.

> Главный принцип методички: ни строки кода, пока проект не описан в документации.
> Спецификация — это ТЗ для AI-агентов: чем конкретнее, тем автономнее сборка.

---

## Шаг 0 — Подготовка (5 минут, без AI)

- [ ] Создай папку проекта, `git init`, пустой первый коммит.
- [ ] Установи/открой Claude Code в этой папке.
- [ ] Реши **одним предложением**, что строишь и зачем (это семя для Слоя 1).

---

## Шаг 1 — Слой 1: PROJECT_IDEA.md (документ идеи)

Это самый важный документ. Пишется для AI, который сгенерирует спецификацию.
**10 обязательных разделов:** Проблема (с числами!) · Решение (пошагово — каждый шаг
станет модулем) · Почему сейчас · Целевая аудитория · Архитектура (диаграмма слоёв +
стек с обоснованием + модули вход→выход) · Монетизация · Конкуренты (таблица) ·
План запуска (MVP отдельно!) · Риски (таблица) · Техдетали (структура репо, таблицы).

**Правило конкретики:** не «продвинутая аналитика», а «график retention по когортам,
данные из X через cron». AI не работает с абстракциями.

Черновик можно надиктовать AI — промпт:

```
Ты — старший продуктовый архитектор. Помоги мне написать PROJECT_IDEA.md по
методологии Spec-First. Моя идея: [1–3 предложения о проекте].
Задай мне МАКСИМУМ 5 уточняющих вопросов (только то, что нельзя додумать), затем
сгенерируй полный PROJECT_IDEA.md со всеми 10 разделами: Проблема (с числами),
Решение (пошаговый процесс, каждый шаг = будущий модуль), Почему сейчас, Целевая
аудитория (основная+вторичная), Архитектура (ASCII-диаграмма слоёв, таблица стека
с обоснованием, таблица модулей вход→выход), Монетизация, Конкуренты (таблица),
План запуска (MVP отдельно от v2, с метриками готовности), Риски (таблица:
риск/вероятность/митигация), Техдетали (структура репозитория, ключевые
таблицы/схемы, пайплайн). Пиши конкретно: числа, имена полей, команды. Без
TODO-заглушек. Если проект учебный/портфельный — Монетизацию и Конкурентов заполни
формально и пометь это.
```

**Проверь сам (чеклист методички):** проблема с числами ✓ решение пошагово ✓
диаграмма слоёв ✓ стек обоснован ✓ MVP выделен ✓. Сохрани как `PROJECT_IDEA.md`,
закоммить.

## Шаг 2 — Слой 2: SPEC.md (техническая спецификация)

Спека НЕ пишется руками — генерируется из документа идеи. Каждый модуль — **6 блоков**:
User Stories · Модель данных · API · Экраны · Бизнес-логика · Крайние случаи.

Промпт (вставь в Claude Code, PROJECT_IDEA.md уже в папке):

```
Роль: старший продуктовый архитектор.
Прочитай PROJECT_IDEA.md. Сгенерируй файл SPEC.md — полную техническую
спецификацию, покрывающую ВСЕ модули из раздела «Решение».

Формат — для каждого модуля ровно 6 блоков:
1) User Stories — минимум 3–5, «Как [роль], я хочу [действие], чтобы [результат]»;
   основной сценарий + крайние случаи.
2) Модель данных — таблицы/схемы: имя, тип (как в SQL: uuid, text, integer,
   timestamptz…), constraints, связи. [Если БД нестандартная — опиши в формате
   этой БД: JSON-схемы, коллекции и т.п., как задано в PROJECT_IDEA.]
3) API и серверная логика — эндпоинты: метод + путь + тело + ответ + коды ошибок
   (формат: POST /api/x — {…} → 201 | 400 | 409).
4) Экраны и компоненты — что отображают, действия, состояния (загрузка, ошибка,
   пусто, успех).
5) Бизнес-логика — валидации, ограничения, формулы, переходы состояний.
6) Крайние случаи — конкретные сценарии с ожидаемым поведением (нет данных, API
   не отвечает, конкурентное изменение…).

Правила: максимум конкретики (типы как в SQL, пути как в коде); без TODO-заглушек;
НЕ задавай вопросов — если ответ есть в PROJECT_IDEA.md, бери оттуда; если ответа
нет — прими разумное решение сам и пометь его комментарием «(допущение)».
Workflow: прочитай PROJECT_IDEA.md целиком → сгенерируй SPEC.md → перепроверь, что
модули стыкуются между собой (общие сущности названы одинаково).
```

**Проверь сам:** у каждого модуля все 6 блоков ✓ типы полей как в SQL ✓ у API есть
коды ошибок ✓ нет TODO ✓. Прочитай спеку глазами 20–30 минут — это твой чертёж,
исправить слово тут дешевле, чем модуль потом. Закоммить.

## Шаг 3 — Слой 3+4: Конфигурация (CLAUDE.md, субагенты, rules, skills)

Генератор превращает спеку в файлы для Claude Code. Промпт:

```
Роль: инженер по конфигурации AI-агентов.
Прочитай PROJECT_IDEA.md и SPEC.md. Сгенерируй пакет конфигурации:

1) CLAUDE.md — МАКСИМУМ 120 строк: обзор (2–3 строки), стек, архитектура (где что
   лежит), правила проекта (безопасность, запреты, стиль), команды запуска/тестов.
   Никаких деталей модулей — они в SPEC.md.

2) Классификация: есть ли в продукте AI-агенты/LLM-интеграции? Если ДА — добавь
   субагента ai-agent-architect; если НЕТ — не добавляй.

3) .claude/agents/*.md — 3–6 субагентов под ЭТОТ проект (например:
   database-architect, backend-engineer, frontend-developer, qa-reviewer).
   Формат каждого: YAML-заголовок (name, description «когда использовать», tools,
   model: opus для архитектуры/безопасности/сложной логики, sonnet для UI/рутины/
   ревью) + тело: Роль, Принципы, Паттерны, Чеклист перед завершением, Интеграция.
   ЖЁСТКОЕ ПРАВИЛО: qa-reviewer получает только Read, Bash, Glob, Grep — БЕЗ
   Write/Edit; он описывает проблемы, не правит.

4) .claude/rules/*.md — 3–5 контекстных правил с glob-паттернами (например:
   правила безопасности для app/Models/**, правила стиля для src/components/**).

5) .claude/skills/*.md — 2–3 навыка для повторяющихся задач (например:
   implement-feature по SPEC_TEMPLATE, create-migration).

6) SPEC_TEMPLATE.md — шаблон описания новой фичи (Описание, User Stories, Модель
   данных, API, Экраны, Бизнес-логика, Крайние случаи, Приоритет/Зависимости).

Все файлы выдай ПОЛНОСТЬЮ, copy-paste ready, без заполнителей «вставьте сюда».
Создай их сразу в нужных путях. Не пиши код приложения.
```

**Проверь сам:** CLAUDE.md ≤120 строк ✓ qa-reviewer без Write/Edit ✓ не все агенты
на opus ✓ rules с glob ✓. Если CLAUDE.md раздуло — попроси сократить, детали в
rules/SPEC. Закоммить.

## Шаг 4 — Каркас

```
По CLAUDE.md и SPEC.md создай структуру папок проекта и минимальный работающий
каркас: точка входа + «пустой» запуск (health-check/заглушка главной страницы),
конфиг окружения (.env.example, .gitignore с .env и локальными конфигами).
Код модулей НЕ пиши. После — остановись, я проверю запуск.
```

**Проверь сам:** проект запускается одной командой из CLAUDE.md ✓ .env в .gitignore ✓.
Закоммить: `scaffold`.

## Шаг 5 — Слой 5: Автономная сборка (модуль за модулем)

Твой цикл на КАЖДЫЙ модуль (порядок — из «Плана запуска» в PROJECT_IDEA):

```
Реализуй модуль «[ИМЯ]» строго по его разделу в SPEC.md и правилам CLAUDE.md.
Используй подходящих субагентов. После реализации: добавь тесты на ключевую
логику модуля и ОСТАНОВИСЬ — я проверю перед следующим модулем.
Не трогай другие модули. Не коммить и не пушь.
```

Затем ТЫ: **запустил → потыкал руками → прогнал тесты → сам закоммитил** →
следующий модуль. После MVP — деплой, README (скриншоты, запуск, «What this
demonstrates»), и только потом v2.

Полезные промпты по ходу:
- Ревью: «Выступи как qa-reviewer: проверь [безопасность/крайние случаи из SPEC].
  Только опиши проблемы списком с приоритетами — ничего не правь.»
- Непонятный код: «Объясни простыми словами, что делает этот участок и почему так —
  мне отвечать за этот код на интервью.»
- Дрейф от спеки: «Сверь реализацию модуля X с SPEC.md, перечисли расхождения,
  ничего не правь.»

## Антипаттерны (из методички — не делай этого)

- ✗ Код без спецификации → фрагменты, переделка через день.
- ✗ Абстрактная спека («продвинутая аналитика») → нужны таблицы/поля/эндпоинты.
- ✗ CLAUDE.md 300+ строк → максимум 120; детали в rules/skills/SPEC.
- ✗ Все субагенты на opus → opus для сложного, sonnet для рутины (в 2× дешевле).
- ✗ qa-reviewer с правами Write/Edit → начнёт «чинить» вместо ревью.
- ✗ Плоский дамп файлов в корень → структура создаётся ДО сборки.
- ✗ Ожидание полной автономии → автономность в качестве решений; контроль,
  запуск и коммиты — на тебе.

## Финальный чеклист перед сборкой

□ PROJECT_IDEA.md: проблема с числами, решение пошагово, диаграмма, стек обоснован,
  MVP отдельно
□ SPEC.md: каждый модуль ×6 блоков, типы как в SQL, API с кодами, без TODO
□ Конфигурация: CLAUDE.md ≤120, qa-reviewer read-only, rules с glob
□ Репозиторий инициализирован, .env настроен и в .gitignore, структура создана,
  Claude Code запущен

> «Качество спецификации определяет качество автономной сборки.
>  Каждый час в документации экономит день разработки.»
